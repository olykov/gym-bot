# Concept — живая запись сета + progressive overload («борьба с собой»)

> Дата: 2026-06-11. Статус: **approved 2026-06-12** (решения оператора — см. §6).
> Задачи: `tasks/progression/` (GYM-130…GYM-137).
> Контекст: оператор видит PR и прошлую тренировку, но при записи «нет задора делать
> больше». Цель — чтобы каждый сет ощущался событием, а прогрессия (progressive
> overload) была видимой целью прямо в момент записи, без превращения апки в игру.

## 0. Принципы (guardrails)

1. **Это логгер, не игра.** Один accent, никакого конфетти/библиотек анимаций, всё за
   `prefers-reduced-motion`, максимум один оркестрированный момент на действие
   (язык спеки §9.4 сохраняется).
2. **Геймификация = сравнение с собой вчерашним**, не очки/уровни/бейджи-зоопарк.
3. **Не врать.** Все цифры — реальные данные; никаких сфабрикованных целей.
4. KISS: Фаза 1 — **ноль изменений API** (всё уже есть в `log-context` +
   `training/day`), серверные фичи только в Фазе 2+.

---

## 1. Ключевая идея: «ghost» прошлой сессии в SetLogger

Аналог ghost-car в гонках. `log-context.last_session_sets` уже возвращает
`{set, weight, reps}` прошлой сессии — сейчас он используется только как невидимый
pre-fill. Сделать его **видимым соперником**:

```
 BENCH PRESS                · Chest
 ─────────────────────────────────────────
 TODAY            LAST TIME   (Sora --hint)
 Set 1  100 × 8   100 × 8     =
 Set 2  102.5 × 8 100 × 8     ▲ +2.5kg     ← accent, только что записан
 Set 3   — · —    100 × 7                  ← ghost: цель следующего сета
 ─────────────────────────────────────────
 SET 3                        PR 102.5 × 5
```

- Recap превращается в **двухколоночное сравнение**: сегодня vs прошлая сессия,
  строка к строке по номеру сета. Ghost-строки — `--hint`, полупрозрачные.
- Следующий непобитый ghost-сет — это **цель, видимая до нажатия Save**: юзер садится
  на скамью и уже видит «прошлый раз: 100×7 — побей».
- После Save строка получает **дельту**: `▲ +2.5kg` / `▲ +1 rep` (accent),
  `=` (hint), `▼ −2.5kg` (hint, без красного — не наказываем за лёгкий день).
- Сортировку recap вернуть ASC (Set 1 сверху): сравнение строк к строкам читается
  только сверху вниз; «последний сет виден» решается auto-scroll к активной строке.
  (Обсудить: это меняет GYM-101.)

Данные: **уже есть, ноль API-изменений.** Дельта = `session set N` vs
`last_session_sets[N]`.

## 2. Момент Save — хореография (детали UI в доке 01 §2)

Последовательность на успешный `POST /training` (суммарно ~700ms, не блокирует ввод):

1. 0ms — haptic success (есть).
2. 0–240ms — новая строка recap: fade + rise + флэш фона (`pr-flare`-язык).
3. 0–180ms — дельта-бейдж «▲ +2.5kg» вкатывается в строку (translateX 4px + fade).
4. 200–800ms — кнопка: «✓ SET 2 — 102.5×8» → морф обратно в «SAVE SET 3».
5. Заголовок SET N тикает (ролл цифры).
6. PR-beat (есть) поверх: pulse чипа + flare строки. Усилить **только** для PR:
   короткий full-width баннер в шите «NEW PR · 102.5kg» (Bebas, accent, 1.2s,
   slide-down → fade) — это самый редкий и самый заслуженный момент.

Reduced-motion: всё мгновенно, haptics остаются.

## 3. Финал сессии — «Done» перестаёт быть молчаливым

Сейчас Done просто закрывает шит. Вместо этого — **session summary** (один экран в том
же шите, body-swap как Phase A↔B):

```
 DONE · CHEST DAY
 12 sets · 3 exercises · 5,840 kg volume
 ▲ 4 sets beat last session   · 1 PR
 Week streak: 7 →
 [ CLOSE ]
```

- Считается **полностью клиентски** из sessionSets + last_session_sets (+ summary из
  кэша для стрика). Ноль API.
- Volume (Σ weight×reps) — первый шаг к «тоннажу» как метрике прогресса.
- Если сессия пустая (открыл-закрыл) — никакого summary, просто закрытие.
- Тон — сдержанный итог, не «🎉 amazing job!!!».

## 4. Прогрессия за пределами сессии (Фаза 2 — нужен API)

1. **e1RM (Epley: `w × (1 + reps/30)`)** — честная метрика силы, когда вес не растёт,
   а повторы растут. Считать на лету в Progress (клиентски из exercise-progress) +
   третий режим графика «e1RM» в SegmentedControl. PR-типы расширяются:
   weight PR (есть) / rep-PR при том же весе / e1RM PR — каждый со своим
   калибром празднования (e1RM PR ≈ тихий чип, weight PR ≈ баннер).
2. **`GET /analytics/exercise-trend?muscle&exercise`** (новый, маленький):
   `{last_session_volume, prev_session_volume, e1rm_trend: [{date, e1rm}]}` —
   питает дельту «вся сессия vs прошлая» в summary и спарклайн в SetLogger-шапке
   (микро-график e1RM за 8 недель рядом с PR-чипом — «куда идёт кривая»).
3. **Dashboard**: карточка «vs last week» (volume/sets по мышцам) вместо или рядом со
   StatCards; DayCard в History получает accent-маркер «PR day».
4. **Overload-streak** (опционально, обсудить): счётчик сессий подряд, где упражнение
   прибавило (вес/повторы/e1RM). Это самый «игровой» элемент — вводить только если
   §1–§3 не дадут достаточного эффекта; риск — мотивация ломаться на deload-неделях
   (стрик обнуляется незаслуженно). Если делать — с «заморозкой» на deload.

## 5. Бот (минимально, для паритета)

Бот остаётся вторым клиентом записи. Дёшево и сердито: в подтверждение записи сета
добавить ту же дельту текстом — `Записал: 102.5×8 (▲ +2.5 к прошлой сессии)` —
данные у API уже есть (log-context). Не строить в боте ghost-таблицы — Mini App
теперь основной интерфейс записи.

## 6. Фазирование и обсуждаемые вопросы

| Фаза | Состав | API | Усилие |
|---|---|---|---|
| 1 | Ghost-сравнение + дельты (§1), Save-хореография (§2), session summary (§3) | нет | M–L (~3–5 дней) |
| 2 | e1RM-режим графика, multi-type PR, exercise-trend, спарклайн (§4.1–4.2) | 1 endpoint | M |
| 3 | Dashboard «vs last week», PR-day маркеры, overload-streak (§4.3–4.4) | 1–2 endpoint | M |
| — | Бот: дельта в подтверждении (§5) | нет | S |

**Решения оператора (2026-06-12) — locked:**
1. **Recap ASC** — да, откатываем DESC из GYM-101 ради построчного сравнения
   (последний сет остаётся видимым через auto-scroll к активной строке).
2. **Дельта: вес приоритетен, при равном весе — повторы.**
3. **Session summary — делаем**, с жёстким ограничением: НЕ удлинять опыт записи.
   Дизайн-следствия: summary появляется только при ≥1 записанном сете, закрывается
   одним тапом В ЛЮБОМ месте (вся поверхность — dismiss), никаких обязательных
   кнопок/шагов, рендер из кэша (ноль ожидания сети).
4. **Overload-streak — отложен** (задача не создаётся; вернуться после замера Фазы 1).
5. **PR-баннер — пробуем** в сдержанном варианте §2.6, смотрим на ощущение.
